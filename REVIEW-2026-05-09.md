# Codebase Review: ChaosPatch
**Date:** 2026-05-09
**Stack:** Next.js 16 (App Router) ┬Ę React 19 ┬Ę Tailwind v4 ┬Ę Neon Postgres (raw SQL) ┬Ę Clerk v7 ┬Ę MCP SDK ┬Ę Vercel
**Project type:** PWA + MCP server (multi-user dev patch tracker)

## Executive Summary

ChaosPatch is a tight, well-scoped product ŌĆö the MCP surface is rich (16 tools), the dashboard/insights pages are polished, and the Tailwind v4 design tokens are coherent. The biggest issues are **two real authorization bugs** in the patch mutation paths (single-patch endpoints don't scope by user, so a signed-in user could mutate another user's patches if they learn a UUID) and **stale documentation** that contradicts the current architecture (README still describes stdio MCP + `middleware.ts`). After those, the next layer is a handful of UI inconsistencies ŌĆö landing-page palette ignores the ADHDesigns tokens, priority badges hardcode dark-mode-only colors, and `<Link>` wraps `<li>` in two places (invalid HTML).

## High Priority Findings

### 1. Patch mutation endpoints don't scope by `userId`
- **What:** `updatePatchStatus`, `addNote`, `updatePatch`, `reopenPatch`, `deletePatch`, and `getPatchById` (`lib/queries.ts:140-202, 220-238`) all execute `UPDATE/DELETE patches WHERE id = $1` with no join/filter on `projects.user_id`. The REST endpoint `app/api/patches/[id]/route.ts` and the MCP tools `cp_start_patch` / `cp_complete_patch` / `cp_add_note` / `cp_delete_patch` / `cp_update_patch` / `cp_reopen_patch` all hit those queries directly. The only check is "are you signed in / is your token valid?" ŌĆö there is no check that the patch belongs to *this* user.
- **Why it matters:** Any authenticated user who learns or guesses a patch UUID can edit or delete it. Patch IDs are UUIDv4 so this isn't trivially exploitable, but it's a real horizontal privilege escalation. `batchUpdatePatches` already does this correctly (joins `projects` and filters by `user_id`) ŌĆö that's the pattern.
- **Suggested action:** Add a `userId` parameter to each single-patch mutation and use the same `UPDATE ŌĆ” FROM projects WHERE projects.user_id = $userId AND patches.project_id = projects.id AND patches.id = $patchId` shape. Have the `RETURNING` clause produce zero rows when ownership fails, and surface a 404 from the route. Also fix `cp_update_project` / `cp_delete_project` paths (they already filter by user ŌĆö verify the pattern is consistent).
- **Effort estimate:** Half-day (mechanical, but touches every patch mutation + tests).

### 2. README and AGENTS.md contradict the actual MCP architecture
- **What:** `README.md` describes the MCP server as a stdio process you point at `mcp/index.js` with `DATABASE_URL` (lines 73ŌĆō85), shows `middleware.ts` in the project structure (line 127), and lists 6 of the 16 MCP tools. The real server is an HTTP route at `app/mcp/route.ts` using `WebStandardStreamableHTTPServerTransport` with OAuth + bearer-token auth, the middleware lives at `proxy.ts`, and there are now 16 MCP tools. `.claude/CLAUDE.md` also says "SSE transport" in Common Issues ŌĆö also wrong (Streamable HTTP). The standalone `mcp/index.ts` file referenced by README does not appear to exist as a working entrypoint (only `app/mcp/route.ts` is present).
- **Why it matters:** Anyone trying to install ChaosPatch from the README will fail. The roadmap still lists MVP items as unshipped. This is the highest-traffic doc surface.
- **Suggested action:** Rewrite the "Connect the MCP server" section around the OAuth-discoverable HTTP endpoint at `https://chaospatch.adhdesigns.dev/mcp` + `/settings` token flow. Replace the tools table with the full 16-tool list. Update project-structure block (`proxy.ts`, no top-level `mcp/`). Strike completed roadmap items. Update `.claude/CLAUDE.md` "SSE" ŌåÆ "Streamable HTTP".
- **Effort estimate:** Quick fix (under an hour).

### 3. Landing page ignores the ADHDesigns design tokens
- **What:** `app/page.tsx` is built entirely on `from-blue-600 via-purple-600 to-pink-600` gradients, `text-gray-700` headings, and `bg-blue-50/purple-50/pink-50` backgrounds ŌĆö none of which reference the `--adhd-*` tokens defined in `globals.css`. Feature card titles use `text-gray-700` with no dark variant, so they go nearly invisible against a dark `slate-900/50` card in dark mode.
- **Why it matters:** This is the only public-facing page, and it's the page that should *most* express the ADHDesigns brand. Right now it looks like a Vercel template.
- **Suggested action:** Rebuild the landing on the ADHDesigns palette (lavender/purple/amber/sage), use the same `bg-card / text-foreground / text-muted-foreground` tokens the rest of the app uses, and fix the dark-mode contrast on feature cards.
- **Effort estimate:** Half-day.

## Medium Priority Findings

### 4. Priority badges hardcode dark-mode-only colors
- **What:** `PRIORITY_STYLES` in both `app/home-content.tsx:359-363` and `app/projects/[slug]/patch-list.tsx:7-11` use `bg-red-950/40 border-red-900` and `bg-yellow-950/40 border-yellow-900` with no light variants. In paper-mode (light theme), these render as near-black blocks against a lavender background.
- **Why it matters:** The light theme is the *primary* theme according to the token mapping ("paper mode"), but the priority badges visually break it.
- **Suggested action:** Replace with token-driven styles ŌĆö e.g., high ŌåÆ `--destructive` (amber), medium ŌåÆ muted yellow that maps to `--warning`, low ŌåÆ muted neutral. Centralize the styles in one shared module since the table is duplicated.
- **Effort estimate:** Quick fix.

### 5. MCP tool input handling skips the documented Zod validation
- **What:** Both `.claude/CLAUDE.md` and `AGENTS.md` say "Validate all inputs with Zod" for MCP tools. `app/mcp/route.ts` declares `type Args = Record<string, string | undefined>` and uses `args.x!` non-null assertions throughout `handleTool`. There is no Zod schema parsing. `cp_batch_update` even has to do a manual `as unknown as` cast because `patch_ids` is an array, not a string.
- **Why it matters:** Malformed MCP requests crash with confusing errors instead of returning structured validation messages, and the `Args` type is actively wrong for the batch endpoint. Low security impact (each handler still hits Postgres-parameterized queries), but it's drift from a stated convention.
- **Suggested action:** Define one Zod schema per tool in a colocated `lib/mcp-schemas.ts`, parse `mcpReq.params.arguments` against the matching schema before dispatch, and return the parser error as the tool's text response when invalid.
- **Effort estimate:** Half-day.

### 6. `<Link>` wraps `<li>` ŌĆö invalid HTML
- **What:** `app/home-content.tsx:379-409` (`PatchRow`) and the equivalent project card link wrap a `<li>` inside `<Link>`. Anchors can't legally contain `<li>` ŌĆö browsers tolerate it but a11y trees and some assistive tech get confused.
- **Why it matters:** Minor a11y / semantics correctness.
- **Suggested action:** Move the `<Link>` *inside* the `<li>` (anchor wraps the inner `div`), or render the row as `<li><Link>ŌĆ”</Link></li>`.
- **Effort estimate:** Quick fix.

### 7. Add-patch empty state references the wrong MCP tool
- **What:** `app/add/add-patch-form.tsx:62-65` shows `cp_add_patch` as the tool to use when the user has no projects. The correct tool there is `cp_add_project`.
- **Why it matters:** Direct contradiction of the user's likely next action when this empty state appears.
- **Suggested action:** One-character fix.
- **Effort estimate:** Quick fix.

### 8. PWA manifest theme/background colors don't match the app
- **What:** `public/manifest.json` uses `theme_color` and `background_color` of `#09090b` (raw slate-950). The actual app background is `--adhd-lavender` (#DBD5E2) in light mode and `--adhd-dark` (#1E1830) in dark mode. The viewport `themeColor` in `app/layout.tsx:26-29` similarly uses `#fafafa` / `#09090b`. Installed PWAs flash a dark-grey splash that doesn't match either theme.
- **Why it matters:** Splash-screen flicker on install, and it's a missed branding moment.
- **Suggested action:** Set both manifest colors to `#1E1830` (or generate a brand splash). Update layout `themeColor` to `--adhd-bg` / `--adhd-dark`.
- **Effort estimate:** Quick fix.

## Low Priority / Nice-to-Have

### 9. Inline edit can't edit notes
- `patch-list.tsx` exposes inline title/priority editing but notes are append-only. Editing notes requires deleting the patch. Consider a "edit notes" mode (full textarea) for completeness.

### 10. No empty/loading states on REST mutations
- `patch-list.tsx` mutation handlers don't catch fetch failures or surface errors ŌĆö a network drop silently rolls the optimistic UI back via `router.refresh()` with no message.

### 11. Search wildcard injection
- `searchPatches` parameterizes the query value but user-supplied `%` and `_` are treated as wildcards. Not a security issue (parameterized), but `"50%"` searches strangely. Escape with `LIKE ŌĆ” ESCAPE` if you care.

### 12. Icon-only buttons missing `aria-label`
- The header theme/insights/settings icons rely on `title` only. Add `aria-label` for screen readers.
